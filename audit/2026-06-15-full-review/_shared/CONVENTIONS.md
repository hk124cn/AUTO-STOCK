# 子报告写作约定

所有子报告使用相同结构，便于汇总。

## 子报告标准结构

```markdown
# 子报告：<主题>

> 范围：列出审查了哪些文件/模块
> 严重程度评级：P0=功能错误/P1=性能或安全/P2=可改进/P3=小问题
> 审查日期：2026-06-14

## 1. 概览
2-3 段说明该模块做什么、整体评价。

## 2. 关键发现（按严重程度降序）
### [P0] 标题
- 位置：file_path:line_number
- 现象：……
- 后果：……
- 证据：code 片段 / 复现命令
- 建议：具体改法

### [P1] 标题
……

## 3. 改进建议（非问题，但有更好做法）
…

## 4. 需要核实的不确定项
…

## 5. 评分（1-5，5 = 优）
- 正确性：?
- 可维护性：?
- 性能：?
- 文档：?
- 总评：?
```

## 命名规范
- 草稿：`raw-notes/notes.md`
- 终稿：`final/SUBREPORT-<短名>.md`，例如 `SUBREPORT-factors.md`
- 证据代码片段直接放子报告内，不要单独存

## 找问题时的切入角度
- 数字处理：0、空、NaN、负值、除零、整数溢出
- 边界：只有 1 只股票、最后一天、休市
- 缓存：命中/失效策略、过期逻辑
- 并发：race condition、文件锁
- 数据缺失：降级还是 fail
- 路径硬编码：可移植性
- 错误吞噬：except: pass / except Exception: ...
- 注释 vs 实际行为不符
- 测试覆盖
- 配置项和默认值
